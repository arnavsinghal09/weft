# Release engineering

How a Weft release artifact is built, what ships with it, and how to verify
you can rebuild it byte-for-byte.

## What ships

- `weft` (CLI) and `libweft_shim.so`, built `--release` (which includes
  debug symbols by design — `profile.release.debug = true`; the replay and
  trace tooling needs them).
- `chord-check`, `chord-trace`, `raft-check` (case-study checkers).
- `sbom/weft-sbom.spdx.json` (SPDX 2.3) and `sbom/weft-sbom.cdx.json`
  (CycloneDX 1.6), generated by `cargo sbom` v0.10.0 from the committed
  `Cargo.lock`. 71 packages (regenerated 2026-07-11 after the stateright oracles landed); every license is permissive and on the
  `deny.toml` allow-list.

License / advisory / source gates: `cargo deny check advisories bans
licenses sources` must be fully green at release time (CI enforces this;
verified green against everything shipped as of 2026-07-11).

## Reproducible build — exact steps

The reference build environment is the CI container image. Reproducibility
was tested empirically on 2026-07-09; results below.

```sh
# 1. Fixed toolchain: rust:1.84-bookworm (rustc 1.84.1 e71f9a9a9).
docker run --rm -it -v "$PWD":/src:ro rust:1.84-bookworm bash

# 2. Fixed build path (this is load-bearing, see caveat below).
mkdir -p /build && cp -r /src/. /build && cd /build

# 3. Build. Cargo.lock is committed; --locked forbids resolution drift.
cargo build --release --locked --workspace

# 4. Hash.
sha256sum target/release/weft target/release/libweft_shim.so \
          target/release/chord-check target/release/raft-check
```

## Measured results (2026-07-09, rustc 1.84.1; re-verified 2026-07-11 on the current tree, aarch64: two from-scratch same-path builds byte-identical)

| condition | result |
|---|---|
| same image, same source, same build path, two from-scratch builds | **byte-identical** (`weft`, `libweft_shim.so`) |
| same image, different build paths | **differ** — expected: `debug = true` embeds the compilation directory (`DW_AT_comp_dir`) in DWARF |
| different paths, `RUSTFLAGS="--remap-path-prefix=<path>=/src --remap-path-prefix=$HOME/.cargo=/cargo"` | **byte-identical** |

So: **the build is reproducible** under a pinned toolchain provided you
either build at the canonical path (`/build`, as above) or remap paths.

## Honest caveats — what is *not* yet guaranteed

1. **Path sensitivity is the release's responsibility, not the build
   system's.** Nothing in the repo pins the build path or sets
   `--remap-path-prefix` automatically; a release script should adopt the
   remap flags into `.cargo/config.toml`'s release profile before the first
   binary artifact is published. Until then, "rebuild at `/build`" is a
   convention documented here, not an enforced invariant.
2. **Toolchain pinning is by container tag, not by hash.** `rust:1.84-bookworm`
   is mutable; a future re-tag (new rustc point release, new glibc) would
   change bytes legitimately. A published release should record the image
   digest (`docker inspect --format '{{index .RepoDigests 0}}'`) alongside
   the artifact hashes. Not yet done because no binary artifact has been
   published yet.
3. **Cross-machine determinism of the build was tested only within one
   host's Docker.** Different CPUs should not matter for `--release` without
   `-C target-cpu=native` (we don't set it), but this was not empirically
   verified across machines.
4. `cargo install weft-dst` from crates.io (once published) will not be
   bit-reproducible against the container build — different rustc, paths,
   and host. The reproducibility story applies to the container recipe
   above.

## Versioning

Surface-compatibility contracts (scenario DSL, weft-log, CLI) are defined in
[../VERSIONING.md](../VERSIONING.md). The weft-log format version is
independent of the crate version and readers must reject unknown versions.
